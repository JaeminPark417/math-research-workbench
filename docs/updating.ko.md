# workbench를 안전하게 업데이트하기

어떤 프레임워크 업데이트보다 연구 자료가 더 중요합니다. 새 배포판을 현재 연구
폴더 위로 끌어다 놓고 **모두 바꾸기** 또는 **전체 덮어쓰기**를 선택하지 마세요.
릴리스 ZIP은 특정 시점의 복사본일 뿐이며, GitHub 템플릿으로 만든 개인 연구
저장소도 원본 템플릿의 업데이트를 자동으로 받지 않습니다.

## 업데이트하기 전에

1. 새 릴리스의 변경 설명을 읽습니다. 필요한 변경이 없다면 업데이트를 미뤄도 됩니다.
2. Markdown, TeX 같은 글 파일에 최근 백업이 있는지 확인합니다.
3. 외부 저장소나 `files/`에 둔 PDF와 다른 큰 파일도 별도로 백업되었는지 확인합니다.
   Git이 제외한 파일과 노트에 링크만 된 첨부 파일은 GitHub 백업에 포함되지 않습니다.
4. 비교하는 동안 파일이 바뀌지 않도록 Obsidian을 닫거나 편집을 멈춥니다.
5. 새 릴리스를 현재 workbench 밖의 **별도 임시 폴더**에 다운로드하고 압축을 풉니다.

`.harness/local.yaml`에는 개인 컴퓨터의 경로가 들어 있을 수 있습니다. 공개 이슈나
릴리스에 복사하지 마세요.

## 권장 업데이트 요청문

기존 workbench 폴더를 로컬 Codex에서 열고 다음과 같이 요청합니다.

```text
별도 폴더에 받은 새 릴리스로 Math Research Workbench를 안전하게 업데이트해줘.
먼저 두 버전을 비교해줘. ideas, papers, notes, projects, files,
.harness/local.yaml, 개인 Obsidian 설정은 바꾸지 마.
교체하려는 프레임워크 파일을 모두 나열하고 예상되는 효과를 설명해줘.
아직 파일을 바꾸지 말고 내 승인을 기다려줘.
```

Codex가 요청할 때만 새 릴리스 폴더에 접근을 허용하세요. 홈 폴더 전체나 클라우드
드라이브 전체가 아니라 그 폴더 하나만 선택합니다.

보통 프레임워크 파일에는 `AGENTS.md`, `.agents/skills/`, `meta/`, `scripts/`,
`docs/`, 배포된 서식 파일이 포함됩니다. 연구 내용에는 보통 `inbox/`, `ideas/`,
`papers/`, `notes/`, `projects/`, `files/`가 포함됩니다. 릴리스 안내에 연구 노트의
형식(schema)이 바뀌었다고 적혀 있다면, 승인하기 전에 기존 노트가 어떻게 바뀔지
미리 보여 달라고 요청하세요.

Codex가 여러 변경을 제안하더라도 한 번의 승인이 모든 작업에 대한 허락은 아닙니다.
파일 목록과 차이를 확인한 다음, 한 번에 한 작업씩 적용하도록 승인하세요. 새 프로그램
설치, 외부 서비스 쓰기, 공개 범위 변경처럼 영향이 다른 작업에는 각각 별도의 승인이
필요합니다.

## GitHub를 사용하는 경우

GitHub 백업은 글 파일의 이전 버전을 비교하고 복원하는 데 도움이 되지만
**자동 동기화는 아닙니다.** 변경을 저장(commit)하고 올리기(push)까지 끝내야
GitHub에 반영됩니다. 또한 템플릿으로 만든 연구 저장소에는 독립된 변경 이력이
생기므로 원본 템플릿과 자동으로 합쳐지지 않습니다.

업데이트 결과를 GitHub에 올리기 전에 다음을 확인하세요.

- 연구 저장소의 공개 범위가 여전히 비공개(private)인지 확인합니다.
- 포함될 파일 전체와 변경 내용(diff)을 검토합니다.
- 노트에 연결된 이미지나 첨부 파일 중 Git이 무시하는 파일을 목록으로 확인합니다.
  그런 파일은 GitHub에 포함되지 않으므로 별도로 백업합니다.
- 강제 올리기(force push), 변경 이력 다시 쓰기, `git reset --hard`,
  `git clean`을 사용하지 않습니다.
- 큰 파일과 PDF는 Git 저장소 밖에 둡니다.

업데이트 중 충돌이 생기면 멈추고 Codex에게 두 버전의 차이를 쉬운 말로 설명해
달라고 하세요. 연구 파일에 **모두 수락**을 선택하지 말고 영향을 받는 내용을
하나씩 검토합니다.

## 확인하고 되돌릴 사본 남기기

승인한 업데이트를 적용한 뒤에는 workbench의 비파괴 점검을 실행하고 경고를
요약해 달라고 요청하세요. 연구 노트 몇 개를 직접 열어 링크, 수식, Obsidian이
정상적으로 작동하는지 확인합니다.

업데이트 전 백업과 새 릴리스 임시 폴더는 적어도 한 번의 평소 연구 세션을 무사히
마칠 때까지 보관하세요. 문제가 생기면 그 백업이나 Git 변경 이력에서 **영향받은
파일만** 되돌립니다. workbench 전체를 삭제하지 마세요.

정리할 때도 연구 파일을 영구 삭제하지 않습니다. 필요 없는 사본이나 정리 대상은
무엇인지 확인한 뒤 보관(archive) 위치로 옮기고, Codex가 제안한 삭제는 대상과
복구 방법을 먼저 설명받은 다음 별도로 승인하세요.
